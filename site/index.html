The problem

The limit always hits at the worst time

Claude Code on the Pro plan enforces a 5-hour rolling window and a weekly cap. When you hit it, the CLI prints Claude usage limit reached. Your limit will reset at <time> and blocks every prompt until the window resets — often mid-task, overnight, or while you're away.

⏳

Work stalls

A long refactor or migration freezes the moment the limit lands — and waits for you to come back and retype.

🌙

You lose the night

The window resets at 3 AM. Nobody is awake to press resume, so hours of potential progress evaporate.

🧠

You lose context

Restarting by hand risks losing the conversation thread — and typing "continue" yet again gets old fast.

How it works

From limit to resume, in five steps

Host

Runs the real claude CLI inside a true embedded terminal — typing, arrow keys, colors and resize all work, on Windows, macOS and Linux.

Detect

Watches the output for the usage-limit message — and only that — so a normal exit never triggers a false wait.

Parse

Turns "reset at 3:00 PM (America/New_York)" into an absolute, timezone-correct moment; if it can't, it falls back to interval polling.

Wait

Re-checks the clock continuously, so the countdown stays accurate across laptop sleep, and the wait is saved so it survives a full restart.

Resume

At reset (plus a safety buffer) it types the resume prompt — continue by default — into the live session (relaunching claude --continue first if it died), verifies the resume took, and retries with backoff if it's still limited.

FAQ

Questions, answered

Does this bypass or evade Claude's usage limits?

No. Claude Keeper only waits for the legitimate reset and then resumes. It never circumvents the limit, and it has no access to the Anthropic API — it simply observes the terminal output of the official CLI.

Do I need API keys or to log in again?

No keys. You authenticate with the claude CLI exactly as you normally would; Claude Keeper just hosts that session in a terminal.

What happens if my laptop sleeps or restarts during the wait?

The wait state is persisted to disk. On wake or relaunch, Claude Keeper reloads it, checks it's still fresh and that the saved command still matches your settings, and resumes on the original schedule. A tick-based scheduler keeps the countdown accurate across sleep.

How does it resume, exactly?

It types the resume prompt (continue by default) plus Enter into the same still-open session, one minute after the stated reset — exactly what you would do by hand. Only if the CLI died mid-wait (or the app was restarted) does it relaunch claude --continue first, then type the prompt.

Will it falsely trigger on a normal exit?

No. The detector matches only the usage-limit message, so a clean, user-driven exit is treated as idle and never starts a wait.

Which platforms are supported?

Windows, macOS, and Linux (x64 and arm64). Prebuilt installers for every platform are available on the releases page — just download the one for your OS.

Is it reliable?

Yes. The detection, scheduling, and recovery behavior is hardened and reviewed so it waits for the legitimate reset and resumes dependably — without bypassing limits or touching the Anthropic API.
